# Observation: India Variant тАФ Key Personnel Intelligence (Source Substitutions)

Source: `00-inbox/pipeline-architecture-india.md` ┬з6, corresponds to US doc ┬з6 ([obs-key-personnel-intelligence](../obs-key-personnel-intelligence.md)).

## Activity

Classify team members into the same 7-tier role taxonomy as the US pipeline, but source the underlying research (regulatory-compliance, forensic, credentials checks) from India-specific registries instead of US ones.

## Inputs

- Same 7-way role taxonomy and classification mechanism as US ┬з6: `key_principal` тЖТ `fund_principal` тЖТ `firm_leadership` тЖТ `advisor` тЖТ `former_member` тЖТ `extended_team` тЖТ `misc`. **Unchanged.**
- Same fail-open defaults: no research context тЖТ `extended_team`; parse failure/exception тЖТ `key_principal`. **Unchanged.**
- Same tiered execution DAG structure (`firm_leader`/`key_person`/`extended_team` depth tiers, same task-count-per-tier design). **Unchanged.**

## Outputs

Same 7-bucket classification output as US, feeding the same tiered research depth. The only change is which sources back 3 of the 10 underlying research-task categories:

| Research category | US source | India source |
|---|---|---|
| `regulatory-compliance` | FINRA BrokerCheck / Form U4-U5 | NISM certification registry (mandatory for AIF/PMS/IA personnel) + SEBI's own individual-registration lookup |
| `forensic-regulatory` | SEC/FINRA/CFTC enforcement search | SEBI adjudication-order search by individual name + NCLT/IBC search (personal insolvency/director-disqualification) |
| `credentials` | CFA/CAIA/CPA, bar admissions, FINRA/FCA licenses | CFA/CAIA still apply globally; add NISM Series modules (mandatory for India-registered fund personnel) + ICAI (Chartered Accountant) for CFO/compliance-officer roles |
| `generic`, `employment-history`, `reputation`, `governance`, `performance`, `oba-conflicts` | тАФ | **Unchanged** тАФ Jina/Exa web research works the same regardless of jurisdiction |

## Systems

- Same `verify-key-principals.ts`-equivalent classification mechanism as US.
- Same Jina/Exa dispatch mechanics as US тАФ only the *grounding targets* (which registries/domains to search) differ per the substitutions above.
- Same codegen source-of-truth pattern as US (`mix research.generate_dag` writing to both `config/research_dag.json` and `src/config/research-dag.ts`) тАФ presumed to extend to India task variants, not explicitly confirmed in source doc.

## People / Actors

- Fully automated, same as US pipeline.

## Timing

- `[UNKNOWN]` тАФ no timing changes noted; presumed same cost/latency profile as US since only the search-target domains change, not the dispatch mechanism.

## Problems / Gaps / Workarounds

- NISM certification registry and SEBI individual-registration lookup are asserted as direct substitutes for FINRA/CFA-style checks, but **no acquisition mechanics are specified** for either (unlike ┬з5's regulatory diligence, which details per-source acquisition strategy) тАФ these may inherit the same "search-and-match, not bulk API" pattern as SEBI's other registries, but this isn't confirmed.
- Same underlying design risk as the US pipeline's fail-open bias applies here тАФ no India-specific change noted, but also no India-specific validation that the fail-open defaults are still appropriate for the different regulatory-scrutiny sources.

## Open Questions
