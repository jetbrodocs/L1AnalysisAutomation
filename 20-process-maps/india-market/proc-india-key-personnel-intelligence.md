# Process: Key Personnel Intelligence — India Source Substitutions

Built from: [obs-india-key-personnel-intelligence](../../10-observations/india-market/obs-india-key-personnel-intelligence.md). Sub-process of step 5.2 in [proc-india-deal-analysis-pipeline](proc-india-deal-analysis-pipeline.md). Companion to [../proc-key-personnel-intelligence.md](../proc-key-personnel-intelligence.md) (US 7-tier classification + research DAG).

## Process Overview

- **Purpose**: Classify team members into the same 7-tier role taxonomy as US, sourcing 3 of the 10 underlying research-task categories from India-specific registries instead of US ones.
- **Trigger**: Same as US — team roster available, step not skipped.
- **End condition**: Same as US — every person bucketed into one of 7 tiers, feeding the same tiered research depth.

## Roles Involved

- Fully automated, same as US.

## Inputs and Outputs

- **Input**: Same team roster sources as US (deck extraction + website scraping + filings).
- **Output**: Same 7-bucket classification as US; underlying research grounded in India sources for 3 categories.

## Process Steps

1. Same classification mechanism and 7-way taxonomy as US (`key_principal` → `misc`), same fail-open defaults (no context → `extended_team`; parse failure → `key_principal`). **Unchanged.**
2. Same tiered execution DAG structure (`firm_leader`/`key_person`/`extended_team`, same task-count-per-tier). **Unchanged.**
3. **Source substitution (decision point) — applies to 3 of 10 research-task categories:**
   - `regulatory-compliance` → NISM certification registry (mandatory for AIF/PMS/IA personnel) + SEBI individual-registration lookup, instead of FINRA BrokerCheck/Form U4-U5.
   - `forensic-regulatory` → SEBI adjudication-order search by individual name + NCLT/IBC search (personal insolvency/director-disqualification), instead of SEC/FINRA/CFTC enforcement search.
   - `credentials` → CFA/CAIA still apply globally; add NISM Series modules + ICAI (Chartered Accountant) for CFO/compliance-officer roles.
4. All other categories (`generic`, `employment-history`, `reputation`, `governance`, `performance`, `oba-conflicts`) run identically to US — Jina/Exa web research is jurisdiction-agnostic.
5. Result feeds `personResearchWorkflow` — same as US, see [proc-india-fund-deep-research](proc-india-fund-deep-research.md)'s neighboring process, [../proc-people-deep-research.md](../proc-people-deep-research.md) for the unchanged research-execution mechanics.

## Systems and Tools

- Same `verify-key-principals.ts`-equivalent mechanism as US.
- Same Jina/Exa dispatch mechanics — only grounding targets differ per the 3 substituted categories.

## Known Issues

- **No acquisition mechanics specified** for NISM registry or SEBI individual-registration lookup (unlike §5's regulatory diligence, which details per-source acquisition strategy) — these may inherit the same "search-and-match, not bulk API" pattern as SEBI's other registries, but this isn't confirmed.
- No India-specific validation noted for whether the fail-open bias (default to `extended_team` on missing context, `key_principal` on parse failure) is still the right default given different regulatory-scrutiny sources.

## Open Questions

- Is there a bulk-searchable NISM registry API, or does this require the same scrape/search-based acquisition burden as SEBI's other registries?
- Should the research-task DAG be one graph with market-conditional sources, or a fully separate India DAG generated by the same codegen step (`mix research.generate_dag`)?
